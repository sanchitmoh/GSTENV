Day 2
Data generator — create all synthetic GST data
Goal: realistic fake invoices + GSTR-2B data saved as JSON files
1
Write environment/data/generator.py Core file
This file generates all your fake data. It needs three functions:

generate_gstin(state_code) — returns a fake but validly formatted GSTIN like 27AABCU9603R1ZX. Format: 2-digit state + 10 chars + 1 digit + Z + 1 check char.

generate_invoice() — returns a dict with keys: invoice_id, supplier_gstin, buyer_gstin, invoice_date, taxable_amount, cgst, sgst, igst, hsn_code, item_description. Use random amounts between 5000 and 500000.

generate_gstr2b(invoice_list, missing_rate=0.3) — takes your invoice list and randomly removes 30% of them (simulating suppliers who didn't file). Also randomly alters amounts in 10% of remaining entries. Returns the modified list as the "government view".
2
Generate and save the sample datasets
At the bottom of generator.py, add a main block that runs and saves three files:
environment/data/invoices_easy.json — 10 invoices, 2 missing from GSTR-2B
environment/data/invoices_medium.json — 20 invoices, amount mismatches + missing
environment/data/invoices_hard.json — 50 invoices, all mismatch types mixed

Run: python environment/data/generator.py to create these files. Check they look like real JSON before moving on.
3
Write environment/rules/gst_rules.py
This file encodes the actual GST law as Python logic. You need two functions:

check_itc_eligibility(invoice, gstr2b_record) — if invoice is missing from GSTR-2B, return "ineligible". If amount differs by more than 20%, return "partial". Else return "eligible". This IS Rule 36(4) simplified.

get_recommended_action(status) — maps each status to an action string: ineligible → "Follow up with supplier to file GSTR-1", partial → "Claim only matched amount, reverse the rest", eligible → "Claim full ITC".
Day 3
Pydantic models + core environment class
Goal: the typed data structures and the main env.py with step/reset/state working
1
Write environment/models.py Spec requirement
Define these Pydantic classes — every field must be typed, no plain dicts:

class Invoice(BaseModel):
  invoice_id: str
  supplier_gstin: str
  taxable_amount: float
  cgst: float; sgst: float; igst: float
  invoice_date: str; hsn_code: str

class GSTObservation(BaseModel):
  task_id: str
  purchase_register: list[Invoice]
  gstr2b_data: list[Invoice]
  current_matches: list[dict]
  unresolved_count: int
  step_number: int
  last_action_error: str | None

class GSTAction(BaseModel):
  action_type: str # "match_invoice", "flag_mismatch", "compute_itc", "submit_report"
  invoice_id: str | None = None
  reason: str | None = None
  payload: dict | None = None

class GSTReward(BaseModel):
  total: float
  itc_accuracy: float
  recall_score: float
  action_correctness: float
2
Write environment/env.py — the main class
Create class GSTAgentEnv. It needs these methods:

reset(task_id) — loads the correct JSON file based on task_id ("invoice_match" loads invoices_easy.json), resets all internal state, returns a fresh GSTObservation.

step(action: GSTAction) — processes one agent action. If action_type is "match_invoice", record the match in internal state. If "flag_mismatch", record the flag. If "compute_itc", run the rules engine. If "submit_report", call the grader and end the episode. Returns tuple: (observation, reward_float, done_bool, info_dict).

state() — returns the current GSTObservation without advancing anything. Just a snapshot.
3

obs = env.reset("invoice_match")
print(obs.task_id, obs.unresolved_count)

If it prints without errors, Day 3 is done. If it errors, fix before moving on.

2
Write environment/graders/grader_invoice_match.py
Function grade(agent_matches: dict, ground_truth: dict) -> float:

Both dicts are {invoice_id: "present" or "missing"}. Calculate F1 score — this is just: correct_calls / total_invoices. Return a float between 0.0 and 1.0. This must be DETERMINISTIC — same inputs always give same output. No randomness here.
3
Write Task 2 + its grader the same way
task2_itc_audit.py — max_steps: 12, uses invoices_medium.json, description says "Determine ITC eligibility for each invoice: eligible / partial / ineligible".

grader_itc_audit.py — grade function compares agent's eligibility decisions against what gst_rules.check_itc_eligibility() returns for each invoice. Score = (correct decisions) / (total invoices). Float 0.0–1.0.
4
Test both graders directly Critical check
Call each grader with a perfect answer and verify it returns 1.0. Call with a wrong answer and verify it returns something less. Call the same inputs twice and verify same output both times. If any of these fail, your grader is broken — fix immediately.
Day 5
Task 3 (Hard) + shaped reward function
Goal: the hardest task working + the multi-signal reward that impresses judges
1
Write environment/tasks/task3_full_recon.py
Task 3 description: "Given the full month purchase register and GSTR-2B, produce a complete reconciliation report with total eligible ITC, discrepancy list with supplier GSTINs, and recommended actions for each discrepancy." Uses invoices_hard.json. max_steps: 20. This task requires the agent to call multiple actions before submitting.
2
Write environment/graders/grader_full_recon.py Most important grader
This grader has FOUR components that combine into one score:

itc_accuracy: is the agent's final ITC number within 5% of the correct number? Full mark if yes, proportional penalty if off.

discrepancy_recall: what % of actual mismatches did the agent find? (mismatches_found / total_mismatches)

action_correctness: for each discrepancy the agent found, did they suggest the right action? Check against get_recommended_action().

hallucination_penalty: did the agent invent invoice IDs that don't exist? Each invented ID subtracts 0.1.

Final score: 0.4*itc + 0.3*recall + 0.2*actions - hallucination_penalty. Clamp to 0.0–1.0.
3
Wire reward into env.py
In the step() method, when action_type is "submit_report":
— call the correct grader based on current task_id
— build a GSTReward object with all component scores
— set done = True
— return (final_observation, reward.total, True, {"reward_breakdown": reward.dict()})

For all other action types, return a small intermediate reward (0.05) for each correct match made. This gives the agent partial progress signal throughout the episode, not just at the end.
Day 6
FastAPI wrapper + local testing
Goal: the environment accessible via HTTP endpoints — how HuggingFace Spaces will talk to it
1
Write environment/server.py
Create a FastAPI app with these four routes:

POST /reset — body: {"task_id": "invoice_match"} — calls env.reset(), returns observation as JSON

POST /step — body: GSTAction as JSON — calls env.step(action), returns (observation, reward, done, info)

GET /state — calls env.state(), returns current observation

GET /health — returns {"status": "ok"} — this is what HuggingFace pings to verify your space is alive
2
Run it locally and test every endpoint Must work before Day 7
Run: uvicorn environment.server:app --port 8000

Then open another terminal and test:
curl -X POST http://localhost:8000/reset -H "Content-Type: application/json" -d '{"task_id":"invoice_match"}'

You should get a JSON observation back. Test /step and /state too. If any return errors, fix them now — Day 7 depends on this working.
3
Run openenv validate Pass/fail gate
This is the contest's automated checker. Run: openenv validate . from your project root. It checks your openenv.yaml, your models, and your endpoints. If anything fails, the error message will tell you exactly what to fix. Do not move to Day 7 until this passes cleanly.
